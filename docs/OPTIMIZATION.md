## Flutter (frontend)

### Сборка и размер

- **Release-сборка** всегда используйте для замеров и распространения:
  - `flutter build apk --release` или `flutter build appbundle --release`
  - Размер будет заметно меньше, чем в debug.

- **Анализ размера:**
  ```bash
  flutter build apk --analyze-size
  ```
  Или после сборки: `flutter pub run devtools` и вкладка «App Size».

- **Обфускация** (уменьшает размер и усложняет обратную разработку):
  ```bash
  flutter build apk --obfuscate --split-debug-info=build/symbols
  ```

### Зависимости

- Периодически обновляйте пакеты и удаляйте неиспользуемые:
  ```bash
  flutter pub outdated
  flutter pub get
  ```
- Не подключайте тяжёлые пакеты «на всякий случай».

### Код и производительность

- **Изображения:** сжимайте картинки, для списков используйте `ListView.builder`, а не `ListView(children: [...])` с большим количеством виджетов.
- **Карта:** ограничивайте число одновременно отображаемых маркеров или используйте кластеризацию.
- **Riverpod:** не создавайте лишние провайдеры, перезапрашивающие данные без необходимости.

## Angular (web-frontend)

### Сборка

- Продакшен-сборка: `npm run build` или `ng build --configuration=production`.
- В `angular.json` уже настроены budgets (предупреждение при initial > 2 MB, ошибка при > 3 MB) и `outputHashing: "all"`.

### Уменьшение размера бандла

- **Lazy loading:** тяжёлые модули подгружайте через `loadChildren` (уже используется для power-lines, poles, equipment, substations).
- **Material:** импортируйте только нужные модули (`MatButtonModule` и т.д.), а не весь `@angular/material`.

### Производительность

- **OnPush:** задавайте `changeDetection: ChangeDetectionStrategy.OnPush` у компонентов, где возможно.
- **TrackBy:** в `*ngFor` используйте `trackBy`, чтобы не пересоздавать DOM при обновлении списков.
- **Подписки:** отписывайтесь в `ngOnDestroy` или используйте `async` pipe.
